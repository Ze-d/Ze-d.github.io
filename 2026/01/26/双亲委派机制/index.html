双亲委派机制

Created2026-01-26|Updated2026-01-26
|Post Views:

这份技术总结将从底层原理、运行流程、核心价值以及实战意义四个维度，为你详细梳理 Java 的双亲委派机制（Parent Delegation Model）。

1. 什么是双亲委派机制？

双亲委派机制是 Java 类加载器（ClassLoader）在加载类时遵循的一种层级规则。

核心准则：

当一个类加载器收到类加载请求时，它首先不会尝试自己去加载这个类，而是把这个请求委派给父类加载器去完成。每一层的类加载器都是如此，因此所有的加载请求最终都应该传送到最顶层的启动类加载器中。只有当父类加载器反馈自己无法完成加载（在它的搜索范围中没有找到所需的类）时，子加载器才会尝试自己去加载。

2. 类加载器的层级结构

在 Java 中，类加载器并非通过继承（Inheritance）关系实现父子，而是通过**组合（Composition）**关系来建立层级：

  1. 启动类加载器 (Bootstrap ClassLoader)：使用 C++ 实现，是 JVM 的一部分。负责加载核心类库（如 rt.jar、java.lang.* 等）。
  2. 扩展类加载器 (Extension ClassLoader)：负责加载 Java 的扩展库（lib/ext 目录）。
  3. 应用程序类加载器 (Application ClassLoader)：负责加载用户类路径（ClassPath）上的类，是我们平时开发最常用的加载器。
  4. 自定义类加载器 (Custom ClassLoader)：开发者根据特殊需求（如网络下载、加密逻辑）自行实现的加载器。

3. 工作流程：向上委派，向下加载

类加载的过程可以分为两个阶段：

阶段一：向上委派（寻找是否已加载）

当 AppClassLoader 收到加载请求时：

  1. 它先检查自己是否加载过，如果没有，委派给父类 ExtClassLoader。
  2. ExtClassLoader 检查自己是否加载过，如果没有，委派给父类 BootstrapClassLoader。
  3. 请求一直到达顶层。

阶段二：向下尝试（实际执行加载）

如果顶层加载器也没加载过：

  1. BootstrapClassLoader 尝试在 /lib 目录下搜寻，找到则返回，找不到则“打回”给子类。
  2. ExtClassLoader 收到打回的请求，尝试在 /lib/ext 目录下搜寻，找不到再打回。
  3. AppClassLoader 尝试在 ClassPath 搜寻，如果最后还没找到，则抛出 ClassNotFoundException。

4. 为什么要这么设计？（核心意义）

① 沙箱安全机制（防止核心类被篡改）

假设没有这个机制，开发者写了一个 java.lang.String 类并试图加载。

  • 如果没有双亲委派：系统可能会加载这个“山寨版”String，导致系统崩溃甚至安全漏洞。
  • 有了双亲委派：加载请求最终由 BootstrapClassLoader 处理，它发现自己已经有了官方 String，直接返回，你的山寨类永远不会被加载。

② 避免类的重复加载

因为每次都先询问父类，如果父类已经加载过了，子类就无需重复加载，保证了在 JVM 全局范围内，基础类对象的唯一性。

5. 打破双亲委派

虽然双亲委派很稳健，但在某些复杂架构下，它会成为障碍：

  • Tomcat 隔离机制：一个 Tomcat 里跑两个 Web 应用，它们可能依赖不同版本的同一个 JAR 包（如 Spring 4 和 Spring 5）。为了让它们互不干扰，Tomcat 的 WebAppClassLoader 会先尝试自己加载，打破了双亲委派。
  • SPI 机制 (JDBC)：DriverManager 在核心库（由 Bootstrap 加载），但 MySQL 驱动在用户路径（由 App 加载）。父类加载器管不了子类的类，因此通过“线程上下文类加载器”实现了反向委派。

6. 总结

维度 总结说明
一句话定义 先请示父类，父类搞不定，子类再动手。
核心价值 安全（防篡改）、唯一（防重载）。
应用场景 默认所有 Java 应用。
例外场景 热部署、SPI、容器化隔离。

通过双亲委派，Java 建立了一套有序、安全、高效的等级制度，确保了整个 Java 生态的基石（核心类库）坚不可摧。
