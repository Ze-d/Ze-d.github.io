**“缓存 + 数据库”在业界的常见组合方式**：各自适合什么业务、典型读写路径长什么样、为什么这么做、常见坑在哪里。你可以把它当成一份“缓存与持久化架构模式速查表”。

------

## 1) 读多写少：Cache-Aside（旁路缓存 / Lazy Loading）

**典型场景**：商品详情、用户资料、配置表、文章内容、热点榜单（读极多、写较少）。

**读路径**

```rust
Client -> Service -> Cache GET
                  -> HIT? yes -> return
                  -> no  -> DB SELECT -> return
                          -> Cache SET (TTL)

```

**写路径**

```rust
Client -> Service -> DB UPDATE/INSERT
                  -> Cache DELETE (or UPDATE)

```

**优点**

- 实现简单，最常见
- 缓存命中后 DB 压力巨降

**坑**

- 缓存与 DB 不一致（写后读老数据）
- 缓存击穿/穿透/雪崩（热点key失效、大量空查、同一时刻过期）

**常用治理**

- “写 DB 后删缓存” + 延迟双删
- 负缓存（缓存空值短TTL）
- 热点Key永不过期 + 异步刷新

------

## 2) 强一致或高命中：Read-Through / Write-Through（由缓存代理读写）

**典型场景**：业务希望把“缓存当数据库前置层”，统一入口；或者用托管缓存（如某些KV中间件）做代理。

**路径**

- Read-through：应用只读缓存，缓存 miss 时由缓存层自动回源 DB
- Write-through：应用写缓存，缓存层同步写 DB

**优点**

- 应用逻辑更干净，缓存策略集中
- 一致性比 cache-aside 更容易控制（尤其 write-through）

**坑**

- 写入延迟更高（同步写DB）
- 缓存层成为关键依赖（要高可用）

------

## 3) 写快、读快、最终一致：Write-Behind / Write-Back（缓存写，异步落库）

**典型场景**：计数器、点赞数、播放量、曝光/点击统计、实时积分、在线时长、游戏状态、IoT 上报聚合。

**路径**

- 写：先写缓存（快）→ 后台异步批量落库（可合并、可去重）
- 读：优先读缓存（最新），必要时读DB（历史/对账）

**优点**

- 吞吐高，能吸收写入尖峰
- 可以批量合并写，DB压力可控

**坑**

- Redis 挂了可能丢数据（除非有日志/队列/AOF）
- 落库延迟导致 DB 不是最新
- 需要处理幂等/重放/对账

**你现在这种就属于这个大类**（只是你读目前还全走DB）。

------

## 4) 防止写放大：聚合缓存（Aggregation Cache）+ 物化结果表

**典型场景**：实时指标看板、热力图、报表预计算、TopN、统计维度聚合。

**路径**

- 原始事件 → 缓存聚合（按窗口、按key累加）
- 定时/流式计算 → 写入“结果表”（materialized snapshot/view）

**优点**

- 查询非常快（读结果表）
- 结果表结构稳定，适合前端/BI

**坑**

- 聚合逻辑复杂（窗口、乱序、迟到）
- 结果表可能体量大（比如全量矩阵点）导致写入压力大

------

## 5) 延迟容忍但读要稳：缓存“最新态” + DB “历史态”

**典型场景**：订单状态、物流轨迹、设备在线状态、直播间状态、会话状态。

**设计思路**

- Redis 保存 **当前状态（latest state）**：读多写多，要求快
- DB 保存 **状态变更历史（events/history）**：可追溯、审计、回放

**优点**

- “当前态”查询极快
- “历史态”可追溯，不怕缓存丢

**坑**

- 当前态与历史态需要对齐（从历史重建 current 的能力）
- 状态机幂等、乱序事件处理

------

## 6) 热点极强：热点Key缓存 + 防击穿（SingleFlight/Mutex/Request Coalescing）

**典型场景**：明星商品库存、秒杀商品详情、爆款活动页、热门用户主页。

**核心问题**
同一个 key 在缓存 miss 时会产生“羊群效应”：大量请求同时打DB。

**解决**

- 缓存 miss 只允许一个请求回源（分布式锁/本地SingleFlight），其他请求等结果
- 热点key永不过期 + 异步刷新
- 分片热点（一个key拆N个）

------

## 7) 反作弊/风控：Redis 做实时特征，DB 做离线画像

**典型场景**：登录风控、刷单/刷券、异常行为检测、频率限制。

**路径**

- 实时特征（分钟级/秒级计数、滑窗）放 Redis（INCR/HyperLogLog/ZSet）
- 离线画像/长期统计放 DB 或数仓（天级/月级）

**优点**

- Redis 擅长实时计数与滑窗
- DB/数仓擅长长期沉淀与分析

------

## 8) 搜索/过滤复杂：缓存索引（Redis）+ DB 真值（MySQL）

**典型场景**：附近的人/门店、排行榜、推荐候选集、标签过滤、在线用户集合。

**做法**

- Redis 用 ZSET/SET 维护索引结构（例如在线集合、分数排名、地理位置）
- 查 Redis 得到候选 ID 列表 → 再批量查 DB 拿详情（或详情也缓存）

**优点**

- 查询结构非常快
- DB 只承担详情读取

**坑**

- 索引与DB一致性维护成本
- 候选集过大导致二次查询慢（需要分页/截断/TopK）

------

## 9) 多级缓存：本地缓存（进程内）+ Redis + DB

**典型场景**：超高QPS、微服务网关、配置/字典、热点少量key。

**路径**

- L1 本地缓存命中最快
- miss 才去 L2 Redis，再 miss 才去 DB

**坑**

- 本地缓存一致性（多实例）
- 发布/变更需要广播失效（PubSub/版本号）

------

## 10) 缓存穿透/空查很多：负缓存（Cache Nulls）+ Bloom Filter

**典型场景**：大量随机ID查询、爬虫/恶意请求、用户输入乱查。

**做法**

- 对不存在的数据也缓存一个“空值”（短TTL）
- 或在 Redis/服务内做 Bloom Filter 先判定“是否可能存在”，再决定是否查DB

------

# 一个“按业务特征选模式”的速记

你可以用这几句来快速归类：

- **读多写少** → Cache-Aside / Read-through
- **写多读多、允许最终一致** → Write-behind（缓存写+异步落库）
- **要快照/统计/聚合** → 聚合缓存 + 物化结果表（你现在这个大类）
- **要“当前态秒级读”+“历史可追溯”** → current in Redis + history in DB
- **热点强、miss代价大** → 防击穿/单飞/热点永不过期
- **查询结构复杂（排行/附近/集合）** → Redis 维护索引 + DB 详情
- **随机乱查多** → 负缓存/Bloom
