## 一、 TCP 报文段格式 (The Segment Format)

TCP 的所有功能都封装在 20 到 60 字节的头部（Header）中。理解这些字段是理解 TCP 约定的基础。

| **字段名称**          | **长度**   | **核心作用**                                                 |
| --------------------- | ---------- | ------------------------------------------------------------ |
| **源端口/目的端口**   | 各 16 bits | **多路复用**：确保数据能准确送达某个特定的应用程序。         |
| **序列号 (Seq)**      | 32 bits    | **数据编号**：标记本数据段第一个字节的序号，解决**乱序**问题。 |
| **确认号 (Ack)**      | 32 bits    | **期待接收**：告诉对方下一次我希望收到哪个序号的包，实现**可靠性**。 |
| **数据偏移**          | 4 bits     | 标记头部长度（因为 Options 是变长的）。                      |
| **标志位 (Flags)**    | 9 bits     | **控制状态**：如 `SYN`(同步)、`ACK`(确认)、`FIN`(释放)、`RST`(重置)。 |
| **窗口大小 (Window)** | 16 bits    | **流量控制**：接收方告诉发送方自己还能收多少，防止缓冲区溢出。 |
| **校验和 (Checksum)** | 16 bits    | **数据完整性**：检测传输过程中是否有比特错误。               |
| **可选项 (Options)**  | 0-40 bytes | 扩展功能，如 **MSS**（最大段大小）、**SACK**（选择性确认）。 |

## 二、 连接管理：三次握手与四次挥手

### 1. 三次握手 (Establishing a Connection)

- **过程**：
  1. **Client -> Server (SYN, seq=x)**: 客户端请求连接，同步初始序号。
  2. **Server -> Client (SYN+ACK, seq=y, ack=x+1)**: 服务端确认收到，并请求连接。
  3. **Client -> Server (ACK, ack=y+1)**: 客户端确认收到服务端的请求。
- **深层考量**：
  - **确认收发能力**：确保双方的发送与接收路径都是通畅的。
  - **防止失效请求干扰**：防止网络中滞留的“旧 SYN 包”突然到达服务端，误建连接浪费资源。
  - **同步初始序号 (ISN)**：双方约定好从哪个数字开始编号。

### 2. 四次挥手 (Terminating a Connection)

- **过程**：
  1. **A -> B (FIN)**: A 没数据发了，想关掉发送通道。
  2. **B -> A (ACK)**: B 收到，但 B 可能还有数据没发完。
  3. **B -> A (FIN)**: B 也发完了，请求关掉通道。
  4. **A -> B (ACK)**: A 确认，双方彻底断开。
- **深层考量**：
  - **半关闭状态**：允许一方先结束发送，但仍能接收数据，保证数据完整传输。
  - **TIME_WAIT 状态**：主动关闭方需等待 **2MSL**。一是确保最后的 ACK 能送达（失败可重传）；二是让网络中所有残留包消失，防止干扰新连接。

## 三、 功能解析：需求、逻辑与实现对应表

这是将 TCP 的“约定”转化为“功能”的核心对照表。

### 1. 可靠传输 (Reliability)

- **为什么要有：** 底层 IP 协议是“尽力而为”的，数据包会丢、会错。
- **如何实现：**
  - **确认应答 (ACK)**：利用 Header 中的 `Ack Number`，实行“收到即回复”的约定。
  - **超时重传 (ARQ)**：发送端开启计时器，若 RTO 内未收到 ACK 则重发。
  - **校验和 (Checksum)**：利用 Header 中的校验字段，发现损坏直接丢弃，由重传补位。

### 2. 有序性与去重 (Ordering & Duplication Free)

- **为什么要有：** 网络路径不同会导致“后发先到”，且重传可能产生重复包。
- **如何实现：**
  - **序列号 (Seq Number)**：利用 Header 中的序列号对字节流进行标记。
  - **重组缓冲区**：接收端协议栈实现一个缓存，按 Seq 重新排序，发现序号重复则丢弃。

### 3. 流量控制 (Flow Control)

- **为什么要有：** 防止发送方发得太快，撑爆接收方的处理能力（缓冲区）。
- **如何实现：**
  - **滑动窗口 (RWND)**：利用 Header 中的 `Window` 字段。接收方实时动态调整该值告诉发送方：“我还能吞下多少字节”。这是一个**端到端**的保护机制。

### 4. 拥塞控制 (Congestion Control)

- **为什么要有：** 保护互联网基础架构。如果大家都疯狂发包导致路由器过载，全网性能会雪崩。
- **如何实现：**
  - **状态变量 (CWND)**：发送方在内存里维护一个“拥塞窗口”变量。
  - **四大算法**：
    1. **慢启动**：从小到大指数级试探网络容量。
    2. **拥塞避免**：快到上限时改为线性增长。
    3. **快重传/快恢复**：根据重复 ACK 快速判断丢包并调整速度。

### 5. 全双工通信 (Full-Duplex)

- **为什么要有：** 效率最大化，允许双方同时发信息。
- **如何实现：**
  - **独立序号**：每个连接维护两套 `Seq` 和 `Ack` 计数器，互不干扰。

## 四、 总结：协议栈开发的视角

实现一个 TCP 协议栈，本质上就是在代码里：

1. **写好数据结构**：精准映射 Header 里的每个位。
2. **写好状态机**：管理从 `CLOSED` 到 `ESTABLISHED` 再到 `TIME_WAIT` 的逻辑。
3. **写好缓冲区管理**：处理数据的切分、排序、重组。
4. **写好时钟管理**：驱动超时重传和各种算法的触发。
